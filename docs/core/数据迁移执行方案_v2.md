# 数据迁移执行方案（ortopbitsmdb → myitsm）

**创建时间**: 2026-05-08
**目的**: 将 ortopbitsmdb（137张旧表，含真实数据）迁移至 myitsm（139张新表，优化结构）
**前置**: 两个数据库均已就绪，Python 后端代码 100% 完成

---

## 1. 迁移范围分析

### 1.1 两库对比

```
ortopbitsmdb (旧)  →  myitsm (新)
     137 张表              139 张表
```

### 1.2 表级迁移策略

| 分类 | 数量 | 策略 | 说明 |
|------|------|------|------|
| **A 类：一对一** | ~100 张 | 字段映射 + 直接导入 | 表名相同，字段有增减 |
| **B 类：多对一** | ~10 张 | 合并导入 | 多张旧表合并为一张新表 |
| **C 类：仅旧库有** | ~15 张 | 归档/丢弃 | 迁移日志表、废弃系统表 |
| **D 类：仅新库有** | ~12 张 | 无需迁移 | 新模块表（SLA/MES/IoT等） |

## 4. 特殊数据处理

### 4.1 复合主键 → 代理键
部分表在新结构中使用了自增 Integer PK 替代复合主键（如 tmc21_usergroup、tmc22_userbusityp）。迁移时需：
1. 直接 INSERT，让 `autoincrement` 自动生成新 ID
2. 旧复合键保留为 `UniqueConstraint`

### 4.2 时间戳字段
- 旧库 `gendate` → 新库 `created_at`
- 旧库 `upddate` → 新库 `updated_at`
- 使用 `NOW()` 填充默认值，保留旧时间戳

### 4.3 LargeBinary 字段
`to_dict()` 已增加 bytes 类型处理（PR#6 修复），迁移时直接 COPY 即可。

### 4.4 P0 优化字段（无旧数据）
- `source_type`（TIT10）：默认 `"DAILY"`
- `asset_type`（TMM35）：根据业务规则推断
- `customer_status`（TIT16）：默认 `"ACTIVE"`
- `tmm22_customers_history`：为空，后续业务产生

### 4.5 密码字段
- 旧库 `passwd` 字段迁移到 `tmc13_users.passwd`
- 新库 `password` 列（哈希），首次迁移后需统一重置密码

## 5. 迁移工具与执行方式

### 方案 A：Python 脚本（推荐）
使用现有 SQLAlchemy 模型，逐表读取 ortopbitsmdb、写入 myitsm。

**优势**：可使用 `to_dict()` 方法、自动处理类型转换、事务可回滚、断点续传。

### 方案 B：pg_dump + psql + 转换
导出旧库 SQL → sed/awk 字段转换 → 导入新库。

**劣势**：字段映射需手动维护，出错难定位。

## 6. 校验方案

| 校验项 | 方法 |
|--------|------|
| 行数校验 | `SELECT count(*)` 逐表对比 |
| 字段值抽样 | 随机 10 行，逐列 MD5 对比 |
| 外键完整性 | `SELECT * FROM new_table WHERE fk NOT IN (SELECT pk FROM ref_table)` |
| 业务一致性 | 关键聚合值对比（如库存总量、客户数、单据数） |
