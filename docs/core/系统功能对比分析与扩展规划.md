# 系统功能对比分析与扩展规划

**文档版本**: v1.0  
**创建时间**: 2026-04-27  
**目的**: 对比用户描述的综合运营管理平台愿景与当前 PB 系统实际功能，明确现有能力边界、缺失模块及其优先级规划。

---

## 1. 系统定位对比

| 维度 | 用户描述的愿景 | 当前 PB 系统实际 |
|------|---------------|-----------------|
| 定位 | 融合 MES + ITSM + IoT + 租赁/资产管理的综合运营平台 | **ITSM + 仓储 + 采购 + 销售 + 资产/押金管理** 的设备运营管理系统 |
| 核心 | "设备全生命周期管理"与"客户业务闭环"双驱动 | **"设备流通管理"与"客户服务闭环"** — 侧重设备在客户侧的部署、维护、回收，不涉及设备制造 |
| 内部线 | 采购 → 质检 → **生产** → 入库 | 采购 → 质检 → 入库（**无生产制造环节**） |
| 外部线 | 预计划 → 执行 → 资产/押金 → ITSM → **数据监控** | 预计划 → 执行 → 资产/押金 → ITSM（**无IoT数据监控**） |

## 3. 缺失功能优先级规划

### 3.1 优先级划分原则

- **重构同步开发**：与现有业务紧密关联、数据库已有表结构基础、对现有流程有直接增强作用的功能
- **重构后迭代（近期）**：有明确业务价值、技术实现可控、但与 PB 等价迁移无直接关系
- **重构后迭代（中远期）**：全新技术领域、需要额外基础设施、或业务需求尚未完全明确

### 3.2 详细规划

#### 🔴 Tier-1: 重构同步开发（融入阶段3-4）

这些功能与现有业务数据高度关联，重构时一并开发成本最低、价值最高：

| # | 功能 | 理由 | 建议阶段 | 工作量估算 | 依赖 |
|---|------|------|---------|-----------|------|
| G1 | **SLA 服务级别管理** | TIT01_TIMEPOINT_AREA 已有区域响应时间定义，只需在 ITSM 工单上增加 SLA 计算引擎和达标率统计 | **阶段2补充** | 1周 | ITSM核心（已完成） |
| G2 | **合同管理完善** | THT01_HTGL 表已存在，补充完整 CRUD + 关联客户/发票 + 有效期管理即可 | **阶段4** | 1周 | 客户主数据 |
| G3 | **发票管理完善** | TAC01_FPSK 表已存在（含开票/回款/合同关联字段），补充完整流程 | **阶段4** | 1周 | 合同管理 |
| G6 | **通知/消息系统** | 替换已停运的飞信(TIT22)，提供统一消息发送接口（短信/邮件），ITSM 工单状态变更、SLA 预警等场景需要 | **阶段4** | 1周 | 无 |

> **总增量**：约 4周工作量，融入阶段3-4期间开发。

#### 🟡 Tier-2: 重构后近期迭代（阶段5，重构完成后1-2个月）

这些功能有明确业务价值，但与 PB 等价迁移无直接关系，适合重构完成后作为增值功能快速迭代：

| # | 功能 | 理由 | 工作量估算 | 前置条件 |
|---|------|------|-----------|---------|
| G4 | **租金/费用自动结算** | 有 TMM61_DEPOSIT_POSMODEL（含 RENT_MONEY/SALE_MONEY），需新建结算引擎：按客户+设备+合同期自动生成账单 | 2-3周 | 合同管理(G2)、押金管理完成 |
| G5 | **财务应收应付** | 需全新设计财务模型（应收/应付/科目/对账/折旧），可对接外部财务系统或自建轻量模块 | 3-4周 | G3发票+G4结算 |
| G9 | **客户自助服务门户** | 需新建前端应用（Web/小程序），开放部分 API 给客户侧：自助报修、工单查询、费用查询、服务评价 | 3-4周 | ITSM API（已完成）、用户认证扩展 |

> **总增量**：约 8-11周工作量，建议按 G4→G5→G9 顺序推进。

#### 🟢 Tier-3: 重构后中远期迭代（阶段6+，视业务需求）

这些功能属于全新技术领域或需要额外基础设施支撑，建议在核心系统稳定运行后视业务需求优先级逐步建设：

| # | 功能 | 理由 | 工作量估算 | 技术要求 |
|---|------|------|-----------|---------|
| G10 | **移动端适配** | 仓库盘点、现场维修等移动场景。可采用响应式 Web 或小程序方案，复用现有 API | 4-6周 | 前端开发（Vue/React/小程序） |
| G7 | **生产制造管理(MES)** | 当前系统定位为"设备流通管理"而非"设备制造"。若未来需要管理设备组装/调试/包装流程，需全新 MES 子系统 | 8-12周 | 生产工单/工序/MES领域建模 |
| G8 | **IoT 数据监控** | 需要全新技术栈：MQTT Broker、时序数据库(TimescaleDB/InfluxDB)、实时大屏(Grafana/自建)、报警引擎 | 12-16周 | 物联网中间件、时序数据库、前端大屏 |

> **总增量**：约 24-34周工作量，建议按业务急迫程度逐步启动。G10（移动端）可优先，G7/G8 视业务扩展需要。

## 5. 关键决策建议

### 5.1 重构同步开发的功能（Tier-1）对现有架构的影响

| 功能 | 新增模型 | 新增API端点 | 对现有代码的影响 |
|------|---------|-----------|----------------|
| G1 SLA | SLAConfig, SLARecord | 3-5个 | ITSM Service层增加SLA计算逻辑 |
| G2 合同 | Contract (基于THT01) | 5-7个 | 客户关联+发票关联 |
| G3 发票 | Invoice (基于TAC01) | 5-7个 | 合同关联 |
| G6 通知 | NotificationTemplate, NotificationLog | 3-5个 | 替换TIT22，跨模块复用 |

### 5.2 需要额外技术栈的功能

| 功能 | 额外技术栈 | 是否影响现有架构 |
|------|-----------|----------------|
| G4 结算 | Celery定时任务（自动生成账单） | 新增任务调度层 |
| G8 IoT | MQTT Broker + TimescaleDB + WebSocket | **独立子系统**，通过API与主系统集成 |
| G9 客户门户 | Vue/React前端 + OAuth2 | 新增前端应用，复用后端API |
| G10 移动端 | 小程序/PWA | 新增移动端，复用后端API |

### 5.3 统一身份认证建议

当前系统仅有内部员工认证（TMC13_USERS + JWT）。若后续增加客户自助门户(G9)和移动端(G10)，建议：
- 阶段4 预留多角色认证设计（内部员工 vs 外部客户）
- 阶段5 实现 OAuth2 / SSO 统一认证网关
- 客户端使用独立的 API Key 或 OAuth2 Client Credentials

### 5.4 数据标准化建议（与用户建议一致）

系统已有设备唯一标识体系：
- `TMM43_EID`：设备 SN 码主表
- `TMM43_EID_TRACK`：设备 SN 变更追踪
- `TMM44_POS_R_EID`：POS 设备-EID 关联

此体系是连接仓储、资产、ITSM 和未来 IoT 的关键纽带，重构时需确保 EID 在所有模块间一致引用。

## 6. 总结

| 分类 | 数量 | 说明 |
|------|------|------|
| 已有功能（PB完整实现） | **25个模块** | ITSM 15+ / 仓储 7 / 采购 6 / 销售 4 / 客户资产 6 / 押金 5 / 质检 3 / 考勤 2 / 其他 7 |
| 已有基础待完善（Tier-1） | **4项** | SLA / 合同 / 发票 / 通知 — 约4周，重构同步开发 |
| 全新近期功能（Tier-2） | **3项** | 结算 / 财务 / 客户门户 — 约8-11周，重构后迭代 |
| 全新远期功能（Tier-3） | **3项** | 移动端 / MES / IoT — 约24-34周，按需启动 |

**核心结论**：用户描述的愿景中约 **75%** 的功能在当前 PB 系统中已有实现或数据库基础，通过等价重构 + Tier-1 扩展即可覆盖。剩余 25% 为全新领域（MES/IoT/移动端/客户门户），建议在重构稳定后按业务需求分阶段建设。
