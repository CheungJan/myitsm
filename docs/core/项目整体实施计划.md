## 1. 项目概述

### 1.1 项目背景

将现有 PowerBuilder ITSM 系统重构为 Python + Flask + PostgreSQL 技术栈，同时完成 Oracle 双库（CCGLPDB + LGREPORTPDB）合并为单一 PostgreSQL 数据库。

### 1.2 项目目标

1. **业务等价迁移**：PB 系统全部业务功能平滑迁移至 Python，功能不中断
2. **数据库统一**：Oracle 双库合并为单一 PostgreSQL，消除 DB Link 跨库依赖
3. **架构现代化**：分层架构（API/Service/Repository）、统一状态机、JWT 认证、Pydantic 校验
4. **扩展增强**：在重构过程中同步完成 Tier-1 功能扩展（SLA/合同/发票/通知）
5. **可维护性提升**：代码从 ~18MB PB 精简到 ~2MB Python（预计精简 89%）

### 1.3 技术栈

| 层面 | 技术选型 | 说明 |
|------|---------|------|
| 后端框架 | Flask（应用工厂模式） | 轻量灵活，适合模块化迁移 |
| ORM | SQLAlchemy + Flask-Migrate | 模型定义 + 数据库迁移管理 |
| 数据库 | PostgreSQL | 替代 Oracle，单库架构 |
| 认证 | JWT (HS256) | 无状态令牌认证 |
| 请求校验 | Pydantic BaseModel | 强类型 Schema 校验 |
| 质量门禁 | black/isort/ruff/mypy --strict/pytest/bandit | 6 项全部通过才可合并 |
| 前端框架 | 待定（建议 Vue 3 或 React） | 见第 5 章前端规划 |

## 4. 数据库迁移（Oracle → PostgreSQL）✅ 已完成

> **状态**: 已完成（2026-05-08）| **完整率**: 99.6% | **详案**: `数据迁移执行方案_v2.md` + `数据迁移问题解决报告.md`

### 4.1 实际执行概况

| 项目 | 原计划 | 实际 |
|------|--------|------|
| 源库类型 | Oracle | PostgreSQL（Oracle 已预先导出为 PG） |
| 表数 | 137 → 139 | 一致 |
| 迁移方式 | ora2pg/Python脚本 | Python 双PG连接器 + 动态字段映射 |
| 耗时 | 预估 7-11 周 | 实际 ~4h（开发+执行+调试） |
| 完整率 | — | **99.6%**（3,339,020/3,353,080 行） |
| 完全匹配表 | — | **86 张**（100%） |
| 有差异表 | — | 13 张（均为源库数据质量问题） |

### 4.2 迁移工具

```
app/migration/
├── config.py              # 双PG连接配置（SOURCE_DATABASE_URL + DATABASE_URL）
├── connector.py           # DualConnector（source + target Engine）
├── field_mapper.py        # information_schema 动态同名列映射 + 模糊匹配
├── batch_runner.py        # 5批执行引擎（单连接+ORDER BY+降级逐行）
└── special_handlers.py    # SYS_USER合并 + 密码迁移
migrate_data.py            # CLI入口（--batch/--dry-run/--validate）
```

### 4.3 迁移范围（原计划参考）

| 来源 | 内容 | 表数量（约） |
|------|------|-------------|
| CCGLPDB（业务主库） | 核心业务表：TIT*、TMM*、TWH*、TPC*、TSL* 等 | ~200 张 |
| LGREPORTPDB（零售库） | 门店经营数据、收入产出、合同等 | ~80 张 |
| **合并后** | 单一 PostgreSQL 数据库，消除 DB Link | ~280 张（去重后预计 ~150 张） |

### 4.2 迁移工具选型

| 工具 | 适用场景 | 推荐度 |
|------|---------|--------|
| **ora2pg** | Oracle → PostgreSQL 自动转换 DDL/DML/存储过程 | ★★★★★ 首选 |
| pgloader | 数据加载，支持 Oracle 到 PostgreSQL 实时迁移 | ★★★★ 备选 |
| 自定义 Python 脚本 | 复杂业务逻辑转换、数据清洗、特殊映射 | ★★★ 补充 |

### 4.3 数据类型映射

| Oracle 类型 | PostgreSQL 类型 | 说明 |
|------------|----------------|------|
| VARCHAR2(n) | VARCHAR(n) | 直接对应 |
| NUMBER(p,s) | NUMERIC(p,s) | 精度保持 |
| NUMBER (无精度) | BIGINT 或 NUMERIC | 视业务含义 |
| DATE | TIMESTAMP | Oracle DATE 包含时间 |
| CLOB | TEXT | 大文本 |
| BLOB | BYTEA | 二进制数据 |
| SYSDATE | NOW() | 当前时间 |
| NVL(a,b) | COALESCE(a,b) | 空值替换 |
| DECODE(...) | CASE WHEN ... END | 条件表达式 |
| SUBSTR(s,p,l) | SUBSTRING(s FROM p FOR l) | 子字符串 |
| ROWNUM | LIMIT / OFFSET | 分页 |
| SEQUENCE.NEXTVAL | SERIAL / GENERATED ALWAYS | 自增序列 |
| DB Link (@CCGL_23) | 消除，改为本地查询 | 双库合并后无需跨库 |

### 4.4 迁移步骤

```
第一步：环境准备（1周）
├── 安装配置 PostgreSQL 目标库
├── 安装 ora2pg 工具
├── 配置 Oracle → PostgreSQL 连接
└── 创建目标数据库和 schema

第二步：DDL 迁移（1-2周）
├── 使用 ora2pg 导出表结构（CREATE TABLE）
├── 转换 Oracle 特有语法（DECODE→CASE、NVL→COALESCE 等）
├── 处理序列、约束、索引
├── 与 SQLAlchemy 模型定义交叉校验（已有 138 个模型）
└── 执行 DDL 并验证

第三步：数据迁移（2-3周）
├── 使用 ora2pg 或 pgloader 导出/导入数据
├── 双库数据合并处理：
│   ├── CCGLPDB 全量表迁移
│   ├── LGREPORTPDB 增量表迁移（去重后）
│   └── 跨库引用（@CCGL_23/@CCGL_24）改为本地查询
├── 数据清洗：
│   ├── 字符编码统一（GBK/UTF-8 → UTF-8）
│   ├── 日期格式标准化
│   └── 空值/默认值处理
└── 分批执行，每批验证

第四步：存储过程迁移（1-2周）
├── 清点 Oracle 存储过程/函数/触发器
├── USP_WH_IN / USP_WH_OUT 等核心过程 → Python Service 层
│   （已在阶段3仓储模块完成 Python 等价实现）
├── 简单过程 → PostgreSQL PL/pgSQL（过渡方案）
└── 最终目标：全部业务逻辑在 Python 层

第五步：数据验证（1-2周）
├── 行数一致性校验：每表 COUNT(*) 对比
├── 主键完整性校验：无丢失、无重复
├── 关键字段抽样校验：金额、日期、状态等
├── 业务场景端到端验证：核心流程跑通
└── 输出验证报告

第六步：切换与回滚（1周）
├── 配置 Python 应用连接 PostgreSQL
├── 灰度切换：逐步将流量切到新库
├── 监控异常指标
├── 回滚预案：保留 Oracle 双库只读副本（至少3个月）
└── 确认切换成功后下线旧库连接
```

| 数据迁移 | 2-3 周 | 可与 DDL 部分并行 |

### 4.6 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| Oracle 特有 SQL 语法无法自动转换 | DDL/DML 失败 | 已建立 SQL 映射清单（`PB_TO_PYTHON_SQL_MAPPING.csv`），逐条核验 |
| 双库数据合并时主键冲突 | 数据丢失或重复 | 迁移前分析两库表名交集，制定合并策略（保留/重命名/合并） |
| 字符编码转换导致乱码 | 中文数据损坏 | 迁移前统一检测编码，转换链路全程 UTF-8 |
| 存储过程转换遗漏业务逻辑 | 功能缺失 | 关键存储过程已在 Python Service 层等价实现，迁移时交叉验证 |
| 性能差异（Oracle vs PostgreSQL） | 查询变慢 | 迁移后执行慢查询分析，针对性建索引和优化 |

## 6. 部署与运维方案

### 6.1 部署架构建议

```
                    ┌──────────────┐
                    │   Nginx      │  反向代理 + 静态资源 + SSL
                    │  (80/443)    │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
       ┌──────▼──────┐ ┌──▼──────┐ ┌──▼──────────┐
       │ Flask API   │ │ Flask   │ │ Vue 前端     │
       │ (Gunicorn)  │ │ Worker  │ │ (静态部署)   │
       │  :5000      │ │  :5001  │ │  /dist/      │
       └──────┬──────┘ └──┬──────┘ └─────────────┘
              │           │
       ┌──────▼───────────▼──────┐
       │     PostgreSQL          │
       │     :5432               │
       └─────────────────────────┘
```

### 6.2 部署方式选择

| 方式 | 适用场景 | 推荐度 |
|------|---------|--------|
| **Docker Compose** | 单机部署/测试环境/中小规模 | ★★★★★ 首选 |
| Docker + K8s | 大规模/高可用/弹性扩缩 | ★★★ 后期考虑 |
| 传统部署 | 无容器化基础设施 | ★★ 兜底方案 |

### 6.3 Docker Compose 参考结构

```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_DB: itsm
      POSTGRES_USER: itsm_user
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"

  api:
    build: .
    command: gunicorn -w 4 -b 0.0.0.0:5000 "app:create_app()"
    environment:
      DATABASE_URL: postgresql://itsm_user:${DB_PASSWORD}@db:5432/itsm
      SECRET_KEY: ${SECRET_KEY}
    depends_on:
      - db
    ports:
      - "5000:5000"

  nginx:
    image: nginx:alpine
    volumes:
      - ./frontend/dist:/usr/share/nginx/html
      - ./nginx.conf:/etc/nginx/conf.d/default.conf
    depends_on:
      - api
    ports:
      - "80:80"
      - "443:443"

volumes:
  pgdata:
```

### 6.4 运维要点

| 项目 | 方案 |
|------|------|
| 日志 | Python logging → 文件 + ELK/Loki（可选） |
| 监控 | Prometheus + Grafana（可选），最小化：健康检查 API `/api/v1/health` |
| 备份 | PostgreSQL: pg_dump 每日全量 + WAL 归档增量 |
| SSL | Let's Encrypt 免费证书 + Nginx 配置 |
| CI/CD | GitHub Actions（已有 quality-gate.yml）→ 扩展部署流水线 |
