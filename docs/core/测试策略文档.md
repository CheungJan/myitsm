## 一、测试架构

### 1.1 技术栈

| 组件 | 说明 |
|------|------|
| 测试框架 | pytest |
| HTTP 客户端 | Flask test client（`app.test_client()`） |
| 数据库 | SQLite 内存库（测试配置 `testing`） |
| 认证模拟 | PyJWT 生成测试 Token（固定用户 `T00001`） |
| CI 集成 | GitHub Actions（`.github/workflows/quality-gate.yml`） |

### 1.2 Fixture 设计

| Fixture | Scope | 说明 |
|---------|-------|------|
| `app` | session | 创建 Flask 测试应用，初始化/销毁数据库表 |
| `_rollback` | function (autouse) | 每个测试后自动回滚事务，保证隔离 |
| `client` | function | Flask 测试客户端 |
| `auth_header` | function | JWT 认证头（`Authorization: Bearer <token>`） |

### 1.3 隔离策略

- **数据库隔离**：使用 SQLite 内存数据库（`sqlite:///:memory:`），测试间通过事务回滚隔离
- **应用隔离**：`session` 级别 app fixture，所有测试共享同一应用实例
- **认证隔离**：固定测试用户 `T00001`，token 过期时间设为远未来（`exp: 9999999999`）

## 三、测试命名约定

### 3.1 文件命名

```
tests/test_{module}_api.py
```

例如：`test_itsm_api.py`、`test_warehouse_api.py`。

### 3.2 类命名

```python
class TestXxxAPI:  # 按业务子模块分组
```

例如：`TestMaintenanceDaily`、`TestStockIn`、`TestBillingRuleAPI`。

### 3.3 方法命名

```python
def test_{action}_{detail}(self):
```

例如：
- `test_create_and_get` — 创建后查询验证
- `test_list_with_type_filter` — 带筛选条件的列表查询
- `test_transition_valid` — 合法状态流转
- `test_transition_invalid` — 非法状态流转校验
- `test_get_rule_not_found` — 404 场景

## 四、运行指南

### 4.1 运行全部测试

```bash
pytest
```

### 4.2 运行指定模块

```bash
pytest tests/test_itsm_api.py
pytest tests/test_warehouse_api.py -v
```

### 4.3 运行指定测试

```bash
pytest tests/test_itsm_api.py::TestMaintenanceDaily::test_create_and_get
```

### 4.4 查看覆盖率

```bash
pytest --cov=app --cov-report=term-missing
```

## 五、质量门禁

### 5.1 CI 流水线（GitHub Actions）

```yaml
# .github/workflows/quality-gate.yml
步骤：
  1. black --check      # 代码格式
  2. isort --check      # 导入排序
  3. ruff check         # Lint 检查
  4. mypy --strict      # 类型检查
  5. pytest             # 测试执行
  6. bandit             # 安全检查
```

所有 6 项必须全部通过才允许合并 PR。

### 5.2 Pre-commit 钩子

```yaml
# .pre-commit-config.yaml
hooks:
  - black
  - isort
  - ruff
```

本地提交前自动运行格式化和 lint 检查。

## 六、后续扩展建议

| 优先级 | 建议 | 说明 |
|--------|------|------|
| P1 | 增加单元测试 | 对 Service 层业务逻辑增加独立单元测试（mock Repository） |
| P1 | 增加覆盖率报告 | 集成 pytest-cov 并设置最低覆盖率阈值（建议 ≥80%） |
| P2 | 增加性能测试 | 对关键查询接口（库存、SLA统计）增加基准测试 |
| P2 | 增加 E2E 测试 | 前端开发完成后，增加端到端 Playwright 测试 |
| P3 | 增加压力测试 | 生产部署前，使用 Locust/k6 进行负载测试 |
