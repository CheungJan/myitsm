# 客户主数据（tmm22_customers）字段规范

> **最后更新**: 2026-05-10

## 一、生命周期字段

### 1.1 customer_status（客户状态）

| 码值 | 中文 | 说明 |
|------|------|------|
| TEMP | 临时客户 | 预计划新建，尚未确认 |
| PENDING | 待确认 | 预计划提交，等待审批 |
| ACTIVE | 正式客户 | 预计划完成 / 手工创建确认 |
| INVALID | 已作废 | 预计划取消 / 客户失效 |
| BLACKLIST | 黑名单 | 信用问题禁止交易 |

**存储**: `code_typ='CS'` → `tmm31_syscodes`

**新建默认值**: `ACTIVE`

### 1.2 source_type（来源类型）

| 码值 | 中文 | 说明 | 触发场景 |
|------|------|------|----------|
| PREPLAN | 预计划创建 | 通过门店预计划工单开通 | 预计划 status=1 完成时自动写入 |
| MANUAL | 手工创建 | 后台管理页面手动创建 | `CustomerList.vue` 新增客户默认值 |
| IMPORT | 导入创建 | 批量数据导入 | 批量导入脚本调用时传入 |
| API | API创建 | 第三方系统对接 | 外部 API 调用时传入 |

**存储**: `code_typ='SRC'` → `tmm31_syscodes`

**新建默认值**: `MANUAL`（后台手动创建）/ `PREPLAN`（预计划完成创建）

### 1.3 preplan_id（预计划单号）

**关联规则**: 通过 `tmm22_customers.cust_card` = `plan_cust.custcard` 匹配，取最新的完成计划（`status='1'` 且 `gendate` 最大）。

**plan_cust.plantyp 类型**:

| plantyp | 含义 | 数据量 |
|---------|------|--------|
| 00 | 全新开通 | 4,053 |
| 10 | 磁卡号变更 | 242 |
| 20 | 旧机翻新 | 3,432 |
| 30 | 设备取回 | 1,881 |

**未来自动化**: 预计划流程创建客户时，自动将 `planno` 赋值给 `preplan_id`。

### 1.4 初始化规则

```sql
-- 存量数据初始化
UPDATE tmm22_customers SET customer_status = 'ACTIVE', source_type = 'MANUAL' WHERE customer_status IS NULL;

-- 预计划来源识别（通过 cust_cd 关联，仅 status='1' 的完成计划）
UPDATE tmm22_customers c SET source_type = 'PREPLAN'
FROM plan_cust p WHERE c.cust_cd = p.custcd AND p.status = '1';

-- 预计划单号回填（通过磁卡号 cust_card 关联，取最新完成计划）
UPDATE tmm22_customers c SET preplan_id = p.planno
FROM (
    SELECT DISTINCT ON (custcard) custcard, planno, gendate
    FROM plan_cust WHERE status = '1'
    ORDER BY custcard, gendate DESC
) p
WHERE c.cust_card = p.custcard AND c.source_type = 'PREPLAN';
```

### 1.4 未来业务自动化

以下业务操作应在创建客户时自动设置 `source_type`：

| 业务类型 | source_type | 说明 |
|----------|-------------|------|
| 设备更换 | PREPLAN | 通过 plan_cust 完成更换 |
| 免费更换 | PREPLAN | 通过 plan_cust 免费更换 |
| 全新开通 | PREPLAN | 通过 plan_cust 新开门店 |
| 后台新增 | MANUAL | CustomerList.vue 新增 |
| 批量导入 | IMPORT | 批量导入脚本 |

## 二、设备统计字段

### 2.1 pos_count（有效设备数）

**计算规则**: `SELECT COUNT(*) FROM tmm35_cust_pos_rl WHERE cust_cd = ? AND useflg = '1'`

**说明**: 实时统计，不可手动编辑。后端 `_resolve_customer_refs()` 自动计算。

### 2.2 s_status（设备状态）

| 码值 | 中文 |
|------|------|
| 1 | 正常 |
| 2 | 停用 |
| 3 | 维修中 |
| 4 | 报废 |
| 9 | 其他 |

**存储**: `code_typ='SS'` → `tmm31_syscodes`

### 2.3 POS 软件配置（opersystem / data_base / soft_edition / systemcode）

**字段说明**:

| 字段 | 含义 | 示例值 |
|------|------|--------|
| opersystem | 操作系统 | Linux |
| data_base | 数据库 | Sqlite 3.7.3 |
| soft_edition | 软件版本 | 1.0.0.2 20150601 |
| systemcode | 内核版本 | — |

**数据来源与更新策略**:

| 阶段 | 数据来源 | 更新方式 |
|------|---------|---------|
| 预计划创建 | 生产指令单的软件配置（tpc13_registerdt） | 自动赋值到客户表 |
| 门店开通后 | 设备监控采集（tio02_device_data） | API 直接覆盖 + 更新 `updated_at` |
| 变更追溯 | tmm22_customers_history | 写变更记录（change_type='SW'） |

**设计决策**:
- 客户表存当前最新值，不做历史冗余
- 设备监控采集到的软件信息为准，手工录入不可靠
- 每台设备详情在 `tmm35_cust_pos_rl.sysinfo` / `softinfo` 中
- 变更审计走 `CustomerHistory` 表
- 生产阶段 → 预计划对接 `tpc13_registerdt`（生产注册明细表）
- 运营阶段 → 设备监控对接 `tio02_device_data`（设备数据采集表）

**前端状态**: 编辑表单中已改为只读（disabled），手工保存时排除这 4 个字段。

### 2.4 软件配置同步 API 规范

**端点**: `POST /api/v1/customers/<cust_cd>/sync-software`

**说明**: 设备监控系统或生产系统调用此接口同步软件配置信息。

**请求体**:
```json
{
    "opersystem": "Linux",
    "data_base": "Sqlite 3.7.3",
    "soft_edition": "1.0.0.2 20150601",
    "systemcode": "",
    "source": "device_monitor"
}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| opersystem | string | 否 | 操作系统 |
| data_base | string | 否 | 数据库版本 |
| soft_edition | string | 否 | 软件版本号 |
| systemcode | string | 否 | 内核版本 |
| source | string | 是 | 来源标识: `plan_create`(预计划) / `device_monitor`(设备监控) |

**处理逻辑**:
1. 更新 `tmm22_customers` 对应字段 + `updated_at`
2. 如果值有变化，写 `tmm22_customers_history` 变更记录：
   ```json
   {
       "cust_cd": "<cust_cd>",
       "change_type": "SW",
       "old_value": "<旧值>",
       "new_value": "<新值>",
       "oper_cd": "<source>",
       "oper_date": "now()"
   }
   ```
3. 返回更新后的客户记录

**响应**:
```json
{
    "code": 200,
    "data": { "cust_cd": "...", "opersystem": "Linux", ... },
    "message": "软件配置同步成功"
}
```

**调用方**:
| source 值 | 调用系统 | 触发时机 |
|-----------|---------|---------|
| `plan_create` | 预计划/生产模块 | 生产指令单完成后 |
| `device_monitor` | IoT 设备监控 | 设备上线后首次采集 + 软件变更时 |

**注意事项**:
- 未传入的字段保留原值，不覆盖
- `old_value` 和 `new_value` 在 history 中以 JSON 字符串存储（含全部有变化的字段）
- 前端不可手动修改这 4 个字段（已 disabled）

## 三、行政区域字段

| 字段 | 层级 | 关联表 | 初始化规则 |
|------|------|--------|-----------|
| country_cd | 国家 | tmm02_country | 默认 `191` = 中国 |
| prvn_cd | 省/直辖市 | tmm03_province | 默认 `09` = 上海 |
| city_cd | 城市/区 | tmm04_city | 按客户分类名称匹配区县，无法匹配则留空 |
| town_cd | 区县/街道 | tmm05_town | 需手动选择 |

### 区县匹配规则

1. 客户分类名含区县关键字 → 自动匹配（如"长宁烟草"→长宁区）
2. 地址含"XX区" → 自动匹配
3. 捷强门店名含"（XX）"标注 → 自动匹配
4. 路名推断 → 补充匹配
5. 无法匹配 → 留空（如"铁路烟草"、"上海海烟"）

### 未匹配分类清单

| 分类 | 原因 |
|------|------|
| 铁路烟草糖酒有限公司 | 非区域范围 |
| 贸易中心卷烟经理部 | 非区域范围 |
| 上海烟酒糖酒 | 无具体区县 |
| 上海海烟糖酒有限公司 | 无具体区县 |

## 五、架构约定

1. **码表统一使用 `tmm31_syscodes`**: 所有代码-名称映射通过 `code_typ` 区分，不新建独立码表
2. **后端解析**: `_resolve_customer_refs()` 统一返回 `*_nm` 字段，前端直接使用
3. **编辑下拉**: 前端通过 `fetchSyscodes(type)` 动态获取选项
4. **详情显示**: 直接用 `detailRow.xxx_nm` 显示中文名

## 六、系统参数集成指南

系统参数（`tmc71_sysparm`）是全局配置单例表，后续业务模块按需读取。

### 已生效参数

| 参数 | 接入模块 | 读取方式 |
|------|---------|---------|
| `allowmultilogon` | 认证登录 | `db.session.get(SysParm, "SYSPARM").allowmultilogon` |

### 待接入参数

| 参数 | 应接入模块 | 说明 |
|------|-----------|------|
| `costtype` | 仓储/财务 | 成本核算方式（1=移动加权 / 2=先进先出） |
| `centralwarehouse` | 仓储调拨 | 中心仓库编码 |
| `poinvaliddays` | 采购模块 | 采购计划超期天数 |
| `soinvaliddays` | 销售模块 | 销售单超期天数 |
| `shopbilltype` | 门店/销售 | 门店单据类型差异 |

### 集成模式

```python
# 一行代码读取参数，无需改前端
from app.models.system import SysParm
sp = db.session.get(SysParm, "SYSPARM")
if sp and sp.costtype == "2":
    result = calculate_fifo(...)  # 先进先出
else:
    result = calculate_weighted_avg(...)  # 移动加权
```

### 开发顺序

1. **后端先接入** — 在业务 Service 中读取参数值
2. **前端无需改动** — 管理员已通过系统参数页配置好值
3. **测试验证** — 修改参数值 → 确认业务逻辑变化
