# ITSM 重构项目整体计划与进展（2026-03-11）

## 1. 文档目的

本文件用于固化 PowerBuilder ITSM 向 Python ITSM 重构过程中的：

- 总体目标与阶段规划
- 当前已完成进展与关键结论
- 三方整合后数据库落地方案

后续会话默认以本文件作为统一基线，避免因上下文切换导致信息丢失。

---

## 2. 项目总体目标

将现有 PowerBuilder ITSM 系统，重构为 Python 技术栈的可视化 ITSM 系统，并完成以下核心目标：

1. 业务能力平滑迁移（功能不中断）
2. 数据库模型去冗余、可治理、可演进
3. 兼容层可控收敛，逐步替代历史依赖
4. 建立可回滚、可验证、可审计的迁移路径

## 4. 当前关键进展与结论

### 4.1 已确认事项

- `NETWORK_LINK` 导入方案可用；最终通过 `REMAP_TABLESPACE=CCGL_DATA:CCGL_MIG_DATA` 完成主体落地。

### 4.2 关键口径澄清

- 当前 `279` 张表是**迁移基线库**（Lift-and-Shift 结果），不是“去重重构后的目标库”。
- 图形客户端中看不到表，常见原因是：
  1. 登录用户为 `PDB_ADMIN`（不是 `CCGL_MIG`）
  2. 对象浏览器存在过滤条件
- `impdp` 导入不是“未提交”问题；导入 DDL/DML 由工具管理事务，非手工 `commit` 导致。

## 5. 数据库落地方案（基线库 + 重构库双层）

## 5.1 设计原则

- 先保真再重构
- 兼容层与目标模型解耦
- 迁移过程可核验、可回滚

## 5.2 物理分层建议

### L0 基线层（现状：`CCGL_MIG`）

用途：

- 保留历史对象形态
- 支撑兼容层（P1/P2）验证
- 作为对照与回归基准

策略：

- 最小化结构改动
- 仅进行必要兼容修复

### L1 目标层（建议新 Schema，如 `ITSM_CORE`）

用途：

- 承载去重、标准化后的业务模型
- 服务 Python 重构系统长期演进

策略：

- 按优化字典生成目标 DDL
- 统一主键、状态码、命名与约束策略
- 对“待下线对象”保留短期兼容视图，分批收敛
