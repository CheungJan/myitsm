## 灰度发布策略

### 灰度范围
- **批次 1（小流量）**：5% 流量，选择 2 个试点门店
  - 门店选择：测试环境门店 + 1 个低业务量生产门店
  - 持续时间：2 小时
  - 监控指标：错误率、响应时间、核心业务成功率
  
- **批次 2（中流量）**：30% 流量，扩展到区域级
  - 门店选择：2 个区域的全部门店
  - 持续时间：4 小时
  - 监控指标：同批次 1 + 数据一致性校验
  
- **批次 3（全量）**：100% 流量，全网发布
  - 发布窗口：业务低峰期（建议 22:00-23:00）
  - 持续时间：24 小时监控
  - 监控指标：全量业务指标 + 用户反馈

## 回滚预案

### 回滚触发条件（任一满足即触发）
1. 核心链路错误率超过阈值（批次 1: >0.1%，批次 2: >0.05%，批次 3: >0.01%）
2. 关键接口 P95 超时连续超过阈值（>500ms 持续 5 分钟）
3. 数据一致性校验失败（与 PB 系统结果不一致）
4. 业务方确认出现语义偏差或功能异常
5. 出现安全漏洞或数据泄露风险

### 回滚执行步骤
1. **停止放量**：立即冻结当前批次，停止继续放量
2. **应用回切**：按模块级回滚预案，将应用版本回切到上一版本
3. **数据回滚**：
   - 如有数据写入，执行数据回滚/补偿脚本
   - 复核关键 SQL 结果一致性
4. **验证复核**：
   - 复核核心接口响应正确性
   - 复核关键 SQL 与 PB 系统一致性
5. **通知通报**：通知业务方、项目组回滚完成状态
6. **复盘分析**：形成问题根因分析（RCA），明确修复方案

### 回滚技术方案
- **应用回切**：通过 CI/CD 系统回退到上一个稳定版本
- **数据回滚**：
  - 如有数据写入，使用 `ROLLBACK` 脚本回退变更
  - 如无法回滚，记录补偿数据供后续人工处理
- **路由回切**：如使用网关路由，切换回 PB 系统路由

## 监控与告警

### 监控指标
1. **业务指标**：
   - 错误率（按接口、按门店）
   - 响应时间（P50/P95/P99）
   - 核心业务成功率（门店关闭、免费更换、免责处理等）
   
2. **技术指标**：
   - CPU/内存使用率
   - 数据库连接池使用率
   - 慢 SQL 数量
   
3. **数据一致性**：
   - 与 PB 系统关键 SQL 结果比对
   - 数据写入一致性校验

### 告警规则
- 错误率超过阈值：立即告警

## 风险评估

### 高风险项
1. **数据一致性风险**：新旧系统数据可能不一致
   - 缓解措施：严格的数据一致性校验，小流量验证
   
2. **性能风险**：Python 实现性能可能低于 PB
   - 缓解措施：性能压测，监控响应时间

3. **业务语义偏差**：理解偏差导致功能异常
   - 缓解措施：业务方深度参与验收，小流量充分验证

### 中风险项
1. **回滚失败风险**：回滚脚本可能执行失败
   - 缓解措施：提前演练回滚流程，准备多套回滚方案

2. **监控盲区**：关键指标未覆盖
   - 缓解措施：完善监控指标，增加业务方自定义监控
