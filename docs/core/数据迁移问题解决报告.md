## 2. 遇到的主要问题与解决方案

### 2.1 字段映射：硬编码 → 动态同名列匹配

**问题**: 初始方案按硬编码 900+ 字段映射，但目标库列名（如 `dept_cd`, `user_nm`）与源库（`deptcd`, `usernm`）命名规范不同，大量映射错误。

**解决**: 改为 `information_schema.columns` 动态取两库同名列交集 + 模糊匹配处理 `*cd`↔`*_cd`、`*nm`↔`*_nm` 等重命名模式。SKIP_TARGET_COLUMNS 排除 `created_at`/`updated_at`/`status` 等目标库专属无源列。

### 2.2 NOT NULL 列缺少默认值

**问题**: 目标库有 `created_at`/`updated_at`/`password`/`status` 等 NOT NULL 列无源数据，批量 INSERT 失败。

**解决**: `write_target_rows()` 自动为每行填充 `created_at`=NOW()、`updated_at`=NOW()、`password`=从 passwd 复制、`status`="1"。

### 2.3 CHAR 尾部空格 + VARCHAR 超长

**问题**: 源库 CHAR(n) 类型字段尾部有空格，部分字段值超过目标列 VARCHAR 限制。

**解决**: 插入前对所有字符串 `rstrip()`，并通过 `information_schema.columns.character_maximum_length` 自动截断超长值。

### 2.4 OFFSET 跨连接不一致导致丢行 ⚠️ 最关键

**问题**: `SELECT ... OFFSET X LIMIT 2000` 在每次 `engine.connect()` 时创建新连接，PostgreSQL 无 `ORDER BY` 时物理顺序在不同连接间不稳定，导致 OFFSET 跳过/重复行。`tmm43_eid_track`（102万行）最初仅导入 63.9%。

**解决**: 
1. 第一次修复：加 `ORDER BY` 前三列 → 提升至 94.8%
2. 第二次修复：同时用单连接读取 + `ORDER BY` → 最终 **99.998%**（仅缺 19 行）

```python
# 修复后的读取模式
with self.connector.source.connect() as src_conn:  # 单连接
    while offset < src_count:
        sql = f"SELECT ... FROM {tbl} ORDER BY col1, col2, col3 OFFSET {offset} LIMIT {batch}"
        rows = src_conn.execute(text(sql)).fetchall()
```

### 2.5 UNIQUE 约束冲突 → 批量插入整体失败

**问题**: 源库部分表存在 PK 重复，批量 `executemany` 遇到第一条重复即整体失败。

**解决**: `write_target_rows()` 先尝试批量 INSERT，失败后降级为逐行 INSERT（跳过冲突行）。

### 2.6 `id` 列被误排除导致 FK 断裂

**问题**: `SKIP_TARGET_COLUMNS` 包含 `id`，导致父表 id 被重新自增生成，子表 FK 引用旧 id 无法匹配，`twh22_pos_change_dt` 等表 100% 插入失败。

**解决**: 从 SKIP_TARGET_COLUMNS 移除 `id`，保留源库 ID 值；插入后通过 `setval()` 同步序列。

### 2.7 手动重命名覆盖

**问题**: 部分表列名差异超出模糊匹配能力范围（如 `custcd`→`supp_cd`、`idtyp`→`id_type`）。

**解决**: 在 `MANUAL_RENAME` 字典中为 10 张表定义手动列名映射。

### 4.2 逐表核查 SQL

以下 SQL 在 Oracle 中执行，结果与 PG 源库（ortopbitsmdb）对比。

#### FK 孤儿：twh22_pos_change_dt（1441行）

```sql
-- 子表引用的 operation_id 是否在父表中存在
SELECT COUNT(*) AS orphan_rows
FROM twh22_pos_change_dt dt
WHERE NOT EXISTS (
    SELECT 1 FROM twh21_pos_change pc WHERE pc.id = dt.operation_id
);

-- 看子表引用了哪些不存在的 id
SELECT DISTINCT dt.operation_id 
FROM twh22_pos_change_dt dt
WHERE NOT EXISTS (
    SELECT 1 FROM twh21_pos_change pc WHERE pc.id = dt.operation_id
)
ORDER BY dt.operation_id;

-- 两个库都查父表
-- CCGLPDB:
SELECT COUNT(*), MIN(id), MAX(id) FROM twh21_pos_change;
-- LGREPORTPDB:
SELECT COUNT(*), MIN(id), MAX(id) FROM twh21_pos_change;
```

#### FK 孤儿：tmm61_deposit_dtl（381条）

```sql
SELECT COUNT(*) AS orphan_rows
FROM tmm61_deposit_dtl dtl
WHERE NOT EXISTS (
    SELECT 1 FROM tmm61_deposit dep WHERE dep.custcd = dtl.custcd
);

SELECT DISTINCT dtl.custcd
FROM tmm61_deposit_dtl dtl
WHERE NOT EXISTS (
    SELECT 1 FROM tmm61_deposit dep WHERE dep.custcd = dtl.custcd
) AND ROWNUM <= 20;
```

#### NULL PK：tmm61_deposit_list（6238行 id 全 NULL）


#### 方式2：Python 导 CSV（推荐，编码处理好）

```python
"""导出 Oracle 问题表到 CSV，用于和 PG 源库逐列对比。"""
import csv
import cx_Oracle

TABLES = [
    # (数据库, 表名)
    ("CCGLPDB", "twh22_pos_change_dt"),
    ("CCGLPDB", "twh21_pos_change"),
    ("CCGLPDB", "tmm61_deposit_list"),
    ("CCGLPDB", "tmm61_deposit_dtl"),
    ("CCGLPDB", "tmm61_deposit"),
    ("CCGLPDB", "twh19_asset_c_a"),
    ("CCGLPDB", "tit10_maintenance_liability"),
    ("CCGLPDB", "tmm35_cust_pos_rl"),
    ("LGREPORTPDB", "twh21_pos_change"),   # 零售库也可能有
]

# 替换为实际连接信息
CONNS = {
    "CCGLPDB": "username/password@host:1521/CCGLPDB",
    "LGREPORTPDB": "username/password@host:1521/LGREPORTPDB",
}

for db_name, table_name in TABLES:
    conn = cx_Oracle.connect(CONNS[db_name])
    cursor = conn.cursor()
    cursor.execute(f"SELECT * FROM {table_name} WHERE ROWNUM = 0")
    columns = [d[0] for d in cursor.description]

    output = f"/tmp/oracle_{db_name}_{table_name}.csv"
    cursor.execute(f"SELECT * FROM {table_name}")

    with open(output, "w", newline="", encoding="utf-8") as f:
        writer = csv.writer(f)
        writer.writerow(columns)
        writer.writerows(cursor)

    print(f"{db_name}.{table_name}: {cursor.rowcount} 行 → {output}")
    cursor.close()
    conn.close()
```

#### 方式3：导出为 INSERT SQL（可直接导入 PG）

如果 Oracle 数据比 PG 源库多，直接生成 INSERT 语句补入：

```python
"""Oracle 问题表导出为 INSERT SQL，可直接 psql 执行。"""
import cx_Oracle

conn = cx_Oracle.connect("username/password@host:1521/CCGLPDB")
cursor = conn.cursor()

tables = ["twh21_pos_change", "twh22_pos_change_dt", "tmm61_deposit_list"]

for tbl in tables:
    cursor.execute(f"SELECT * FROM {tbl} WHERE ROWNUM = 0")
    cols = [d[0] for d in cursor.description]
    cursor.execute(f"SELECT * FROM {tbl}")

    with open(f"/tmp/insert_{tbl}.sql", "w", encoding="utf-8") as f:
        for row in cursor:
            vals = []
            for v in row:
                if v is None:
                    vals.append("NULL")
                elif isinstance(v, (int, float)):
                    vals.append(str(v))
                else:
                    escaped = str(v).replace("'", "''")
                    vals.append(f"'{escaped}'")
            f.write(
                f"INSERT INTO {tbl} ({', '.join(cols)}) "
                f"VALUES ({', '.join(vals)});\n"
            )
    print(f"{tbl}: {cursor.rowcount} 行")

cursor.close()
conn.close()

# 导入 PG 源库
# psql -U cheungjan -d ortopbitsmdb -f /tmp/insert_twh21_pos_change.sql
```

### 4.5 判断标准

| 情况 | 结论 | 处理 |
|------|------|------|
| Oracle 与 PG 源库一致 | 源数据本身就有问题 | **无需处理**，迁移正确 |
| Oracle 有但 PG 源库没有 | Oracle→PG 合并时丢失 | **重新导出该表**，用 INSERT SQL 补入 PG 源库 |
| Oracle 与 PG 源库都没有 | 可能查错库 | 尝试 LGREPORTPDB 或其他 schema |

## 5. 迁移工具架构

```
app/migration/
├── config.py              # 双库连接配置（SOURCE_DATABASE_URL + DATABASE_URL）
```
