## 2. 遇到的主要问题与解决方案

### 2.1 字段映射：硬编码 → 动态同名列匹配

**问题**: 初始方案按硬编码 900+ 字段映射，但目标库列名（如 `dept_cd`, `user_nm`）与源库（`deptcd`, `usernm`）命名规范不同，大量映射错误。

**解决**: 改为 `information_schema.columns` 动态取两库同名列交集 + 模糊匹配处理 `*cd`↔`*_cd`、`*nm`↔`*_nm` 等重命名模式。SKIP_TARGET_COLUMNS 排除 `created_at`/`updated_at`/`status` 等目标库专属无源列。

### 2.2 NOT NULL 列缺少默认值

**问题**: 目标库有 `created_at`/`updated_at`/`password`/`status` 等 NOT NULL 列无源数据，批量 INSERT 失败。

**解决**: `write_target_rows()` 自动为每行填充 `created_at`=NOW()、`updated_at`=NOW()、`password`=从 passwd 复制、`status`="1"。

### 2.3 CHAR 尾部空格 + VARCHAR 超长

**问题**: 源库 CHAR(n) 类型字段尾部有空格，部分字段值超过目标列 VARCHAR 限制。

**解决**: 插入前对所有字符串 `rstrip()`，并通过 `information_schema.columns.character_maximum_length` 自动截断超长值。

### 2.4 OFFSET 跨连接不一致导致丢行 ⚠️ 最关键

**问题**: `SELECT ... OFFSET X LIMIT 2000` 在每次 `engine.connect()` 时创建新连接，PostgreSQL 无 `ORDER BY` 时物理顺序在不同连接间不稳定，导致 OFFSET 跳过/重复行。`tmm43_eid_track`（102万行）最初仅导入 63.9%。

**解决**: 
1. 第一次修复：加 `ORDER BY` 前三列 → 提升至 94.8%
2. 第二次修复：同时用单连接读取 + `ORDER BY` → 最终 **99.998%**（仅缺 19 行）

```python
# 修复后的读取模式
with self.connector.source.connect() as src_conn:  # 单连接
    while offset < src_count:
        sql = f"SELECT ... FROM {tbl} ORDER BY col1, col2, col3 OFFSET {offset} LIMIT {batch}"
        rows = src_conn.execute(text(sql)).fetchall()
```

### 2.5 UNIQUE 约束冲突 → 批量插入整体失败

**问题**: 源库部分表存在 PK 重复，批量 `executemany` 遇到第一条重复即整体失败。

**解决**: `write_target_rows()` 先尝试批量 INSERT，失败后降级为逐行 INSERT（跳过冲突行）。

### 2.6 `id` 列被误排除导致 FK 断裂

**问题**: `SKIP_TARGET_COLUMNS` 包含 `id`，导致父表 id 被重新自增生成，子表 FK 引用旧 id 无法匹配，`twh22_pos_change_dt` 等表 100% 插入失败。

**解决**: 从 SKIP_TARGET_COLUMNS 移除 `id`，保留源库 ID 值；插入后通过 `setval()` 同步序列。

### 2.7 手动重命名覆盖

**问题**: 部分表列名差异超出模糊匹配能力范围（如 `custcd`→`supp_cd`、`idtyp`→`id_type`）。

**解决**: 在 `MANUAL_RENAME` 字典中为 10 张表定义手动列名映射。

## 4. Oracle 原始库核查指南

连接 Oracle 生产库后，针对上述问题表执行以下核查 SQL（Oracle 语法）：

```sql
-- 1. 核查 twh22 FK 孤儿
SELECT COUNT(*) FROM twh22_pos_change_dt dt
WHERE NOT EXISTS (SELECT 1 FROM twh21_pos_change pc WHERE pc.id = dt.operation_id);

-- 2. 核查 tmm61_deposit_list id 是否全 NULL
SELECT COUNT(*), COUNT(id) FROM tmm61_deposit_list;

-- 3. 核查 tmm61_deposit_dtl FK 孤儿
SELECT COUNT(*) FROM tmm61_deposit_dtl dtl
WHERE NOT EXISTS (SELECT 1 FROM tmm61_deposit dep WHERE dep.custcd = dtl.custcd);

-- 4. 核查 twh19 PK 重复
SELECT opbillid, COUNT(*) FROM twh19_asset_c_a 
GROUP BY opbillid HAVING COUNT(*) > 1;

-- 5. 核查 twh21_pos_change id 起始值
SELECT MIN(id), MAX(id) FROM twh21_pos_change;

-- 6. 核查各表行数与当前 PG 源库对比
SELECT 'twh22_pos_change_dt' AS tbl, COUNT(*) FROM twh22_pos_change_dt
UNION ALL SELECT 'twh19_asset_c_a', COUNT(*) FROM twh19_asset_c_a
UNION ALL SELECT 'tmm61_deposit_dtl', COUNT(*) FROM tmm61_deposit_dtl
UNION ALL SELECT 'tmm61_deposit_list', COUNT(*) FROM tmm61_deposit_list;
```

**判断标准**：
- 若 Oracle 中同样存在孤儿/NULL/重复 → **源数据质量问题**，非迁移缺陷
- 若 Oracle 中数据完整、与 PG 源库不一致 → **Oracle→PG 合并过程有丢失**，需重新导出该表

## 5. 迁移工具架构

```
app/migration/
├── config.py              # 双库连接配置（SOURCE_DATABASE_URL + DATABASE_URL）
├── connector.py           # 双 PostgreSQL 连接器
├── field_mapper.py        # 动态同名列映射 + 模糊匹配 + 手动覆盖
├── batch_runner.py        # 5 批执行引擎（单连接+ORDER BY+降级逐行）
└── special_handlers.py    # SYS_USER 合并 + 密码迁移
migrate_data.py            # CLI 入口（--batch/--dry-run/--validate）
tests/
├── test_migration_config.py       # 连接器测试
├── test_migration_field_mapper.py # 映射引擎测试
└── test_migration_validation.py   # 逐表行数校验
```

**执行命令**：
```bash
export SOURCE_DATABASE_URL=postgresql://user:pass@host:5432/ortopbitsmdb
export DATABASE_URL=postgresql://cheungjan@localhost:5432/myitsm
uv run python migrate_data.py              # 全量
uv run python migrate_data.py --batch 1    # 单批
uv run python migrate_data.py --dry-run    # 预演

# 校验
SOURCE_DATABASE_URL=... DATABASE_URL=... uv run pytest tests/test_migration_validation.py -v
```
