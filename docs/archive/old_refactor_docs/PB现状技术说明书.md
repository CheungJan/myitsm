# PowerBuilder 现状技术说明书（myitsm）

> 文档版本：v0.2  
> 生成时间：2026-02-27（更新）  
> 适用范围：`e:\project\myitsm\src` 导出的 25 个 PBL 库源码

## 1. 文档目标

本说明书用于：

1. 沉淀当前 PowerBuilder 系统的可维护技术资产；
2. 为“先优化、后重构”提供统一认知；
3. 作为后续 Python 版本（Flask）分阶段替换依据。

---

## 2. 系统总体概览

### 2.1 技术栈（现状）

- 客户端：PowerBuilder（DataWindow + UserObject + Window）
- 数据库：Oracle（SQLCA 事务对象，大量 SQL 直连）
- 资源与配置：INI + 资源文件（图片/图标）

### 2.2 运行结构（逻辑）

1. 应用启动 -> 登录窗口（`w_r_logon`）
2. 登录成功 -> 主窗口（`w_m_appmain`）
3. 通过菜单/按钮动态打开业务对象（`Open/OpenWithParm`）
4. 业务对象内执行查询、编辑、保存、送审、作废、关单

### 2.3 导出源码规模（25库）

- 统计口径：`*.sru/*.srw/*.srd/*.srf/*.srm/*.srs/*.sra/*.srj`
- 核心库：
  - 基础框架：`app_main.pbl`, `app_objects.pbl`, `base_objects.pbl`, `base_srv.pbl`, `base_cust.pbl`, `dw2xls.pbl`
  - 系统管理：`app_system.pbl`, `app_d.pbl`, `app_dddw.pbl`
  - 业务域：`purchase.pbl`, `sale.pbl`, `wh.pbl`, `qc.pbl`, `itsm.pbl`, `itsm02.pbl`, `report.pbl` 等

> 注：本次已结合《pbl文件说明》补齐模块归属，`base_cust.pbl` 与 `dw2xls.pbl` 归为基础能力层。

## 3. 分层与模块划分（PB 视角）

### 3.1 基础层

- `base_objects.pbl`：公共控件/基类（`u_dw`, `u_tab`, `u_sle` 等）
- `base_srv.pbl`：通用服务（日志、配置、时间、转换、登录管理）
- `base_dwsrv.pbl`：DataWindow 扩展服务（排序、筛选、下拉检索）
- `base_attrib.pbl`：属性对象（显示、编辑、日志、返回参数等）
- `base_cust.pbl`：客户/供应商/商品基础资料能力（主数据维护基础模块）
- `dw2xls.pbl`：Excel 导出能力模块（报表/查询结果导出支撑）

### 3.2 应用层

- `app_main.pbl`：主窗口框架、导航、页签模块管理
- `app_objects.pbl`：应用通用对象（按钮、服务对象）
- `app_system.pbl`：用户、用户组、菜单、权限
- `app_d.pbl` / `app_dddw.pbl`：系统管理类 DataWindow 定义

### 3.3 业务层（ERP + ITSM）

- 采购：`purchase.pbl`
- 销售：`sale.pbl`
- 仓储：`wh.pbl`, `wh_syc.pbl`
- 质检：`qc.pbl`
- 报表：`report.pbl`
- ITSM：`itsm.pbl`, `itsm02.pbl`, `new_deposit.pbl`
- 其他：`price.pbl`, `trans.pbl`, `lsgldata.pbl`, `kq.pbl`, `appraisal.pbl`

### 3.4 菜单驱动的业务归类（基于系统菜单导出）

一级菜单可映射为以下业务域（示例）：

- 采购管理（10）-> `purchase.pbl`, `wh.pbl`
- 销售管理（20）-> `sale.pbl`, `itsm02.pbl`, `new_deposit.pbl`
- 质检管理（30）-> `qc.pbl`, `wh.pbl`
- 仓库管理（40）-> `wh.pbl`, `wh_syc.pbl`
- 计划管理（15）-> `sale.pbl`, `new_deposit.pbl`
- ITSM（90）/ITSM报表（91）-> `itsm.pbl`, `itsm02.pbl`, `report.pbl`
- 报表管理（75）/统计查询（70）-> `report.pbl`, `lsgldata.pbl`
- 基础信息（80）-> `base_cust.pbl`, `app_system.pbl`, `price.pbl`, `trans.pbl`
- 系统管理（99）-> `app_system.pbl`, `app_d.pbl`, `app_dddw.pbl`

菜单数据同时反映以下历史状态：

1. 存在“有效/无效”并存菜单，说明历史功能有下线未清理现象；
2. 存在 `*_bak`、`*测试`、`old/new` 并存菜单，说明迭代中保留了大量过渡入口；
3. 存在同类功能在不同部门菜单重复配置（如实施部/市场部数据校验）。

## 4. 关键流程梳理

### 4.1 登录与会话

- `w_r_logon` 完成：用户输入校验、自动登录、错误次数控制
- `nvo_appmanager.of_logon` 调用登录窗并回收登录属性

### 4.2 主窗口模块打开

- `w_m_appmain.oe_openobject` 按对象名动态打开模块
- 使用 Tab 页维护并记录当前模块信息

### 4.3 权限管理

- 用户、用户组、菜单、用户菜单维护完整存在
- 菜单保存后可刷新主界面菜单
- 存在业务用户与系统用户（`sys_user/sys_role_user`）同步逻辑

### 4.4 ITSM 典型工单状态流

- 新建 -> 编辑 -> 保存 -> 作废/关闭
- 关闭过程伴随：业务流水记录、计划回写、关单表落库
- 设备变更等模块还存在存储过程调用（如数据导入）

## 5. 历史问题与技术债

### 5.1 架构性问题

1. **界面层直接写 SQL**：事务边界分散，复用和测试困难；
2. **跨模块耦合高**：对象间通过消息、全局变量、动态事件交互；
3. **状态码分散**：`useflg/current_status` 等规则未形成统一字典。

### 5.2 可维护性问题

1. **复制粘贴开发痕迹重**：部分模块注释与实际业务不符；
2. **命名历史包袱**：对象命名和业务语义存在漂移；
3. **冗余风险**：存在大量疑似低引用对象（需业务确认后判废）；
4. **菜单配置冗余**：存在无效菜单、测试菜单、`bak/new` 并存配置，增加运维与培训成本。

### 5.3 风险问题

1. 动态 SQL 与字符串拼接风险；
2. 存储过程依赖不透明；
3. 权限同步逻辑分散，审计困难。

## 6. 优化优先级（重构前）

### P0（立即）

1. 建立对象台账：对象 -> 菜单 -> 业务域 -> 数据表；
2. 建立状态码字典（含含义、可流转路径、触发事件）；
3. 固化关键流程：登录、工单关闭、计划回写、送审作废。

### P1（近期）

1. 抽取 SQL 清单（按模块与表聚合）；
2. 识别重复实现（采购/销售/质检类似单据流程模板）；
3. 输出冗余候选列表并组织业务确认。

### P2（重构并行）

1. 统一审计日志模型（操作人、时间、动作、前后值）；
2. 统一错误码与异常返回结构；
3. 统一分页、筛选、导出规范。

## 7. Python 重构建议（结论）

### 7.1 语言建议

- **主线建议：Python（Flask/FastAPI）**
- 原因：团队当前能力匹配、迁移速度快、维护成本低、适合内部系统渐进替换。

### 7.2 Java 角色建议

- 可作为后续专项：高并发/高吞吐子域（若未来出现）
- 当前阶段不建议全量转 Java，避免拉长建设周期。

### 7.3 重构策略

1. 共存迁移：PB 与 Python 并行；
2. 模块替换顺序：登录权限 -> ITSM核心 -> 采购/销售/仓储/质检 -> 报表；
3. 每替换一模块，做“功能 + 数据 + 流程”三重回归。
