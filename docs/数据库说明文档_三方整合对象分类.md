- 哪些是三方合并后持续使用的业务对象；
- 哪些是迁移治理/校验/检查用对象；
- 为什么当前 `user_tables=293` 仍大于 L1 目标对象数。

## 3. 对象分类口径

### A. 三方合并后实际业务对象（长期运行）

- 口径来源：`MIG_P3_L1_OBJECT_SET`（`P3_STAGE02/WAVE_01/FROZEN`）
- 对象范围：`TABLE` + `VIEW`
- 用途：三方整合后业务主链路运行对象（KEEP + MIGRATE）
- 数量口径：`134`

> 说明：这是“三方整合后使用哪些表/视图”的唯一主口径。

### B. 迁移治理与运行基础设施对象（非业务主数据）

- 口径来源：`MIG_P3_L0_TRIM_CAND.candidate_type='INFRA'`
- 识别规则（见 `DBA_P3_STAGE05_L0_TRIM_CANDIDATE.sql`）：
  - 对象名匹配 `MIG_%`
  - 对象名匹配 `ITSM_%`
- 用途：迁移台账、流程控制、核验留痕、治理支撑
- 历史数量口径：`27`

### C. 已处置的历史裁剪候选对象

- 口径来源：`MIG_P3_L0_TRIM_CAND.candidate_type='TRIM_CANDIDATE'`

## 4. 检查/校验/治理表清单（重点）

下表均属于治理与过程留痕对象，不属于业务主数据模型：

| 分类 | 代表对象 | 作用 |
|---|---|---|
| P2 回归与切换检查 | `MIG_P2_STAGE_CHECK`, `MIG_P2_REG_RESULT`, `MIG_P2_COMPAT_PLAN` | 记录回归、切换、兼容层下线编排 |

| P3 回滚占位清理与残留收口 | `MIG_P3_RB_CLEAN_CTRL`, `MIG_P3_RB_CLEAN_EXEC`, `MIG_P3_RESIDUAL_ACTION` | 清理回滚占位并形成最终残留处置清单 |

---

## 5. 为什么是 293 张表而不是 134 张

- `134` 是 L1 业务目标对象集（主业务口径）；
- `293` 是当前库内运行态总表数（业务对象 + 基础设施对象 + 治理留痕对象）；
- 因项目采用“快速落地 + 可审计”策略，保留必要治理台账是设计内结果，不代表整合未完成。

---

## 6. 三方合并后实际表/视图清单获取方式

请以以下 SQL 为准导出权威清单：

```sql
SELECT source_object_type,
       source_object_name,
       target_object_name,
       governance_tag,
       freeze_status
FROM MIG_P3_L1_OBJECT_SET
WHERE stage_no = 'P3_STAGE02'
  AND wave_no = 'WAVE_01'
  AND freeze_status = 'FROZEN'
ORDER BY source_object_type, source_object_name;
```
