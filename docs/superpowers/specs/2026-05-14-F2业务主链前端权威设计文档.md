## 3. 推进原则

### 3.1 依赖链递进

单人开发，按依赖链推进，边铺页面边提取可复用模式。

```text
第1步：预计划（1-2天）        → 业务流起点，plan_cust 列表/创建/详情
第2步：仓储（2天）            → 依赖物料/仓库，完成入库/出库/库存/盘点
第3步：采购/生产需求（1.5-2天） → 依赖仓储/物料，完成采购计划/登记/单据/验收入口，并在缺货且需生产新设备/整机时预留生产需求入口
第4步：ITSM 高频工单（3天）   → 依赖预计划/仓储/客户，完成维修/开通/翻新/变更
第5步：ITSM 剩余工单（2天）   → 回收/保养/关店/免费更换/未关单/附表
第6步：补全（2天）            → 详情/审核/批量/状态流转/交互打磨
```

### 3.2 PB源码参考原则

规划和实现过程中，遇到以下细节不确定时，优先参考 `PBsrc/` 中对应 PowerBuilder 模块的原业务逻辑：

- 字段含义与默认值。
- 按钮行为与操作入口。
- 单据状态、审核标志、流转条件。
- 明细录入规则、行号规则、数量/金额口径。
- 查询条件、默认筛选、排序方式。
- 异常提示和业务限制。

PB源码只作为业务细节参考源，不作为照搬对象。原系统存在历史包袱、重复逻辑、硬编码状态、字段语义不清和用户体验不足等问题，最终实现应以后端已重构API、当前核心文档、P0优化方案、数据库当前结构和前端可用性为准。

### 3.3 业务细节确认顺序

1. 优先查当前核心文档和后端API契约。
2. 若仍不明确，再查 `PBsrc/` 对应PB模块确认原系统行为。
3. 若PB行为与P0优化或当前数据模型冲突，以当前优化方案为准。
4. 对关键差异在实现记录或阶段总结中标明。

### 5.6 生产需求与MES联动

预计划确认后，需要先通过库存查询判断是否已有可用成品或整机；若库存不足且业务确认需要生产新的设备或整机，应形成生产需求并传递到MES生产模块。

| 阶段 | 职责 | 页面/接口 | 说明 |
|------|------|-----------|------|
| F2 | 识别生产需求 | 预计划详情、库存查询、采购/生产决策入口 | 标记“需生产”，记录来源预计划、客户、物料/BOM、数量、期望交付时间 |
| F2 | 预留跳转入口 | `/sales/plans/:id`、`/warehouse/stock` | 缺货时提供“生成生产需求/查看生产工单”入口 |
| F3 | 创建生产工单 | `POST /api/v1/mes/work-orders` | 由生产模块承接正式生产计划 |
| F3 | 生产计划与工序 | `/api/v1/mes/work-orders/<wo_id>/processes`、`/api/v1/mes/processes` | 维护工序、进度、责任人和状态 |
| F3 | 物料消耗 | `/api/v1/mes/work-orders/<wo_id>/materials` | 记录BOM/配件消耗 |
| F3 | 完工入库联动 | MES完工 → 仓储入库 `/api/v1/warehouse/stock-in` | 完成新设备/整机生产后进入库存，再支撑ITSM开通/更换 |

设计约束：

- F2不实现完整MES生产排程，但预计划和库存页面必须保留生产需求字段或操作入口。
- 生产需求必须保留来源单据关联，如 `planno`、客户、物料编码、BOM或整机标识、数量。
- F3 MES完成后，应能从生产工单回溯到预计划，并从预计划查看生产进度。
- 完工后的设备或整机应通过仓储入库进入库存，再被ITSM新机开通、设备更换、翻新等流程引用。


## 9. 风险与决策

| 风险 | 当前判断 | 决策 |
|------|----------|------|
| QC写操作不独立开放 | 已有只读列表/详情API，写操作由采购入库流程触发 | F2只做只读列表/详情，不做创建/审核 |
| 预计划触发生产需求 | MES已有接口但完整联动属于F3 | F2记录/展示生产需求入口，F3完成生产计划、工序、完工入库联动 |
| 话务台无独立API | 后端未见calls蓝图 | F2先并入预计划呼出确认体验 |
| 供应商主数据未在F1完成 | 采购依赖供应商 | F2采购阶段补轻量页面或只读选择 |
| 12天周期较紧 | 页面数量多 | 先可用闭环，后补复杂明细和批量 |
| PB源码与P0优化冲突 | 老系统存在历史包袱 | 以当前优化方案为准 |
| API文档与实际代码差异 | 个别端点需复核 | 实现前以 `app/api/*.py` 为准 |

## 10. 开发约定

- 使用 Vue 3 `<script setup lang="ts">`，遵循现有F1代码风格。
- 缩进使用4空格，不使用分号。
- API统一通过 `frontend/src/api/request.ts`。
- 不在前端硬编码业务状态名称，优先读取 `fetchSyscodes()` 或后端解析字段。
- 列表页必须支持加载态、空状态、错误提示。
- 所有新增页面必须接入路由和菜单。
- 所有真实接口调用必须符合 `{ code, message, data }` 响应结构。

## 11. 验收标准

- [ ] 预计划：创建预计划 → 选择客户 → 填写物料/设备信息 → 生成预计划单号。
- [ ] 仓储：入库/出库单列表、详情、创建、审核可用；库存查询可用。
- [ ] 采购：采购计划 → 登记 → 单据主链页面可用。
- [ ] ITSM高频工单：日常维护、新机开通、翻新、设备变更至少完成列表、详情、创建、状态流转入口。
- [ ] ITSM剩余工单：回收、保养、门店关闭、免费更换、未关单至少完成列表和详情入口。
- [ ] 所有列表页支持搜索/筛选/分页。
- [ ] TypeScript 编译通过。
- [ ] 后端API联调通过，未完成接口不得伪造成功逻辑。
- [ ] 最小业务链路可演示：预计划 → 库存查询 → 入库/出库、采购计划或生产需求入口 → ITSM日常维护/新机开通。

### 12.2 F3 与 F2 的衔接点

- F2工单详情页应预留押金、合同、发票、SLA、通知记录等关联入口，但F2阶段不实现复杂联动。
- F2客户、资产、设备、工单页面的主键和路由参数应保持稳定，便于F3按客户、合同、设备、工单反向关联。
- F2通用组件应避免只适配单一工单类型，后续需可复用于押金、合同、财务、门户、SLA等列表页。
- F2实现质检只读列表/详情，后续如需质检创建/审核，可作为F3或单独补充阶段与采购、仓储入库衔接。
- F2预计划和库存决策页面应预留生产需求入口，F3 MES负责生产工单、生产计划、工序进度、物料消耗、完工入库联动。

### 12.3 F3 延后原则

- 不因F3模块提前设计而扩大F2开发范围。
- F2只保证菜单结构、路由命名、组件模式不阻塞F3。
- F3实施前需重新核对后端API、权限菜单和实际业务优先级。
